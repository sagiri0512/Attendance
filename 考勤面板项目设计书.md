## 一、项目概述

### 核心规则

| 功能 | 规则 |
|---|---|
| **打卡** | 每天可多次打卡，**最早时间=上班，最晚时间=下班**，一个接口搞定。**凌晨00:00~05:00打卡归属前一天**（record_date = 昨天） |
| **加班申请** | 提交 → **PL先审** → 通过后推送给PM → **PM再审** → 双方通过后自动生效。标记是否特殊情况/跨天 |
| **打卡变更** | 考勤异常时申请修改时间，流程同加班（PL→PM→自动执行） |
| **请假申请** | 提交 → **PL先审** → 通过后推送给PM → **PM再审** → 双方通过后自动生效。PL请假由PM审，PM请假自动同意 |

### 系统角色与权限矩阵

| 角色 | 取值 | 打卡 | 申请 | 待办审批 | 查看下属 | 员工管理 | 薪资设置 | 绩效设置 | 统计报表 |
|---|---|---|---|---|---|---|---|---|---|
| 普通员工 | 0 | 自己 | 可提交 | 无 | 仅自己 | 否 | 否 | 否 | 个人统计 |
| PL（项目组长） | 1 | 自己 | 可提交 | **本组成员**（加班+变更+请假） | 本组成员 | 否 | 否 | 否 | 部门统计 |
| PM（项目经理） | 2 | 自己 | 可提交 | **本组成员**（加班+变更+请假） | 本组成员 | 否 | 否 | **组绩效** | 部门统计 |
| HR/管理部 | 3 | 自己 | 可提交 | 无（不管审批） | 全公司 | **是** | **是** | **是** | 全公司统计 |
| 系统管理员 | 4 | — | — | **审批PM提交的加班和变更申请**（请假无需系统管理员审批） | 全公司 | 全部 | 全部 | 全部 | 公司统计 | **确认节假日配置** |

> **HR/管理部核心职责**：
> 1. **新增员工**：设置姓名、职级（PM/PL/PG）、基本工资、各项补贴、社保缴费基数、公积金缴费基数、公积金缴费比例（工号自动生成）
> 2. **调整员工**：调整员工工资、职级、社保缴费基数、公积金缴费基数和比例（3.2节 `/api/employee/update`）
> 3. **设置PL/PM**：设置员工的 PL（pl_id）和 PM（pm_id）（3.2节 `/api/employee/update`）
> 4. **绩效管理**：设置/调整员工绩效奖金（或通过设置组绩效批量调整）
> 5. 维护节假日配置（work_calendar），含人工确认
> 6. 设置社保参数（social_insurance_config）
> 7. 确认并执行月工资发放

### 逐级上报规则（请假/加班/变更 通用）

```
普通员工提交申请 → PL 审批 → PM 审批 → 自动生效
      ↓
PL 提交申请       → PM 审批（跳过PL自审）     → 自动生效
      ↓
PM 提交申请       → 自动同意（跳过PL和PM审批）  → 自动生效
      ↓
HR/管理部提交申请  → PL/PM 审批（同普通员工流程） → 自动生效
```

> **说明**：
> - 加班和打卡变更：PM审批后还需系统管理员审批（三级审批）
> - 请假：PM审批后自动生效，**无需系统管理员审批**（二级审批）
> - PM提交任何申请均自动同意（包括请假、加班、变更）

## 二、数据库设计

### 2.1 ER 关系

```
员工表(employee) 自关联 ─── pm_id / pl_id 指向上级
员工表(employee) 1 ─── N 打卡记录(attendance_record)
员工表(employee) 1 ─── N 请假申请(leave_request)
员工工表(employee) 1 ─── N 加班申请(overtime_request)
员工表(employee) 1 ─── N 打卡变更申请(attendance_correction)
员工表(employee) 1 ─── N 组绩效(group_performance) [pm_id]
```

### 2.2 表结构

#### `employee`（员工表，含 pm_id/pl_id 自关联 + 薪资组成）

```sql
CREATE TABLE employee (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '员工ID',
  emp_no VARCHAR(20) NOT NULL COMMENT '工号（自动生成，如EMP0001）',
  password VARCHAR(100) NOT NULL COMMENT '登录密码（BCrypt加密存储）',
  real_name VARCHAR(50) COMMENT '员工姓名',
  pm_id BIGINT COMMENT '所属PM员工ID（自关联employee.id）',
  pl_id BIGINT COMMENT '所属PL员工ID（自关联employee.id）',
  role TINYINT NOT NULL DEFAULT 0 COMMENT '角色：0员工(PG) 1PL 2PM 3HR/管理部',
  phone VARCHAR(20) COMMENT '联系电话',

  /* 薪资组成（管理部设置） */
  base_salary DECIMAL(10,2) DEFAULT 0 COMMENT '基本工资',
  position_salary DECIMAL(10,2) DEFAULT 0 COMMENT '岗位工资',
  performance_bonus DECIMAL(10,2) DEFAULT 0 COMMENT '绩效奖金（PM设置，组共享）',
  housing_subsidy DECIMAL(10,2) DEFAULT 0 COMMENT '房补',
  car_subsidy DECIMAL(10,2) DEFAULT 0 COMMENT '车补',
  meal_subsidy DECIMAL(10,2) DEFAULT 0 COMMENT '餐补',
  other_subsidy DECIMAL(10,2) DEFAULT 0 COMMENT '其他补贴',

  status TINYINT DEFAULT 1 COMMENT '账号状态：0禁用 1正常',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted TINYINT DEFAULT 0 COMMENT '0未删除 1已删除',
  UNIQUE KEY uk_emp_no (emp_no),
  INDEX idx_pm_id (pm_id),
  INDEX idx_pl_id (pl_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='员工表';
```

> **薪资说明**：
>
> - `base_salary + position_salary` 参与缺勤扣款计算（按日薪/时薪折算）
> - `performance_bonus` 不参与扣款，按月固定发放（或按考核结果发放）
> - 各项补贴不参与扣款，固定发放
> - 应发工资 = 基本工资 + 岗位工资 + 绩效 + 各项补贴 + 加班费 - 缺勤扣款 - 请假扣款

#### `attendance_record`（打卡记录表，多次打卡一天一条）

```sql
CREATE TABLE attendance_record (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '打卡记录ID',
  emp_id BIGINT NOT NULL COMMENT '员工ID',
  record_date DATE NOT NULL COMMENT '打卡日期',
  clock_in_time DATETIME COMMENT '上班打卡时间',
  clock_out_time DATETIME COMMENT '下班打卡时间',
  status TINYINT DEFAULT 0 COMMENT '0正常 1缺勤 2请假',
  absent_hours DECIMAL(4,1) DEFAULT 0 COMMENT '缺勤小时数',
  leave_hours DECIMAL(4,1) DEFAULT 0 COMMENT '请假小时数',

  leave_request_id BIGINT COMMENT '关联请假申请ID',        -- 已有
  overtime_record_id BIGINT COMMENT '关联加班记录ID',       -- 新增，可NULL

  remark VARCHAR(200) COMMENT '备注',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_emp_date (emp_id, record_date),
  INDEX idx_record_date (record_date),
  INDEX idx_emp_id (emp_id),
  INDEX idx_leave_request_id (leave_request_id),
  INDEX idx_overtime_record_id (overtime_record_id)         -- 新增索引
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='打卡记录表';
```

**status 与 absent_hours 计算规则：**

| status | 触发条件 | absent_hours 计算方式 |
|---|---|---|
| 0 正常 | 上班不晚于09:00 且 下班不早于18:00 | 0 |
| 1 缺勤 | 上班时长少于8小时 | clock_out_time - clock_in_time - 1 < 8 |

> **注意**：迟到+早退同时存在时，取缺勤小时数累加，status 以迟到优先（也可改为单独状态，按需调整）。

#### `leave_request`（请假申请表，顺序双人审批：PL→PM）

```sql
CREATE TABLE leave_request (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '请假申请ID',
  emp_id BIGINT NOT NULL COMMENT '申请人ID（关联employee.id）',
  leave_type TINYINT NOT NULL COMMENT '请假类型：1事假 2病假 3年假 4调休',
  start_time DATETIME NOT NULL COMMENT '请假开始时间',
  end_time DATETIME NOT NULL COMMENT '请假结束时间',
  days DECIMAL(5,1) COMMENT '请假天数（扣除周末）',
  reason VARCHAR(500) COMMENT '请假原因',

  /* 审批阶段 */
  step TINYINT DEFAULT 1 COMMENT '当前审批阶段：1等PL审批 2等PM审批 3审批完成',

  /* PL 审批 */
  pl_status TINYINT DEFAULT 0 COMMENT 'PL审批状态：0待审 1通过 2驳回',
  pl_approve_time DATETIME COMMENT 'PL审批时间',

  /* PM 审批 */
  pm_status TINYINT DEFAULT 0 COMMENT 'PM审批状态：0待审 1通过 2驳回',
  pm_approve_time DATETIME COMMENT 'PM审批时间',

  /* 最终结果 */
  final_status TINYINT DEFAULT 0 COMMENT '最终状态：0审批中 1已通过 2已驳回 3已取消',

  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '申请提交时间',
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  INDEX idx_emp_id (emp_id),
  INDEX idx_final_status (final_status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='请假申请表';
```

> **审批规则**：
> - 普通员工：PL审批（step 1→2）→ PM审批（step 2→3，自动生效）
> - PL本人提交：跳过PL，直接等PM审批（step=2）
> - PM本人提交：跳过PL和PM，自动生效（step=3, final_status=1）
> - 驳回：PL或PM任一方驳回，final_status=2，流程结束

#### `overtime_request`（加班申请表，顺序双人审批 + 实际时长计算）

> **字段说明**：
> - `step`：当前审批阶段（1=等PL审 2=等PM审 3=已完成）
> - `apply_hours`：**申请时长**（员工填写）
> - `actual_hours`：**实际时长**（系统根据 clock_out_time 自动计算，以实际为准）
> - `salary_multiplier`：工资倍数（1.5/2.0/3.0，系统根据日期自动判定）
> - `is_special/is_cross_day`：特殊标记

```sql
CREATE TABLE overtime_request (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '加班申请ID',
  emp_id BIGINT NOT NULL COMMENT '申请人ID（关联employee.id）',
  overtime_date DATE NOT NULL COMMENT '加班日期',
  apply_hours DECIMAL(4,1) NOT NULL COMMENT '申请加班小时数',
  actual_hours DECIMAL(4,1) DEFAULT 0 COMMENT '实际加班小时数（系统根据打卡记录自动计算）',
  reason VARCHAR(500) COMMENT '加班原因',
  is_special TINYINT DEFAULT 0 COMMENT '是否特殊情况：0普通 1特殊情况',
  step TINYINT DEFAULT 1 COMMENT '当前审批阶段：1等PL审批 2等PM审批 3审批完成',

  /* 新增：从 work_calendar 快照 */
  day_type TINYINT NOT NULL COMMENT '日期类型快照 0工作日 1休息日 2法定假',
  wage DECIMAL(2,1) NOT NULL DEFAULT 1 COMMENT '工资倍数快照 1/1.5/2/3',
    
  /* PL 审批 */
  pl_status TINYINT DEFAULT 0 COMMENT 'PL审批状态：0待审 1通过 2驳回',

  /* PM 审批 */
  pm_status TINYINT DEFAULT 0 COMMENT 'PM审批状态：0待审 1通过 2驳回',
  pm_approve_time DATETIME COMMENT 'PM审批时间',

  /* 最终结果 */
  final_status TINYINT DEFAULT 0 COMMENT '最终状态：0审批中 1已通过 2已驳回',

  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '申请提交时间',
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
```

#### `work_calendar`（工作日历表，替代 holiday_config）

```sql
CREATE TABLE work_calendar (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '记录ID',
  date DATE NOT NULL COMMENT '日期',
  day_type TINYINT NOT NULL COMMENT '日期类型：0工作日 1休息日 2法定节假日 3调休假期',
  holiday TINYINT DEFAULT 0 COMMENT '是否放假：0否（含补班） 1是',
  holiday_name VARCHAR(50) COMMENT '节日名称（如：春节、国庆节）或补班名称（如：元旦后补班）',
  wage TINYINT NOT NULL DEFAULT 1 COMMENT '工资倍数基数：1正常 2双倍 3三倍',
  after TINYINT DEFAULT NULL COMMENT '补班方向：NULL=非补班 0节前补班 1节后补班',
  target VARCHAR(20) COMMENT '补班关联的节日名称（如：春节、国庆节）',
  confirmed TINYINT DEFAULT 0 COMMENT '人工确认状态：0待确认 1已确认',
  confirmed_by BIGINT COMMENT '确认人ID（role=4系统管理员）',
  confirmed_at DATETIME COMMENT '确认时间',
  source VARCHAR(10) DEFAULT 'api' COMMENT '数据来源：api自动同步 manual手动录入',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  UNIQUE KEY uk_date (date),
  INDEX idx_day_type (day_type),
  INDEX idx_confirmed (confirmed)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='工作日历表';
```

> **字段说明**（对齐 timor.tech 节假日 API 数据格式）：
>
> | API字段 | work_calendar字段 | 说明 |
> |---------|------------------|------|
> | `holiday` | `holiday` | `true`=放假，`false`=补班 → 存为 0/1 |
> | `name` | `holiday_name` | 节日名称（如"春节"）或补班名称（如"元旦后补班"） |
> | `wage` | `wage` | 工资倍数基数：1=正常，2=双倍，3=三倍 |
> | `after` | `after` | `true`=节后补班，`false`=节前补班 → 存为 1/0/NULL |
> | `target` | `target` | 补班关联的节日名称 |
> | — | `day_type` | 综合判定：0工作日(holiday=false) 1休息日(周末无API记录) 2法定假(wage=3) 3调休假(wage=2且holiday=true) |
> | — | `confirmed` | 系统管理员(role=4)人工确认后设为1，未确认的记录不计入工资计算 |
>
> **day_type 自动映射规则**：
>
> | 条件 | day_type | 含义 |
> |------|----------|------|
> | API `holiday=false`（补班日） | 0 | 工作日 |
> | 周六日且无API记录 | 1 | 正常休息日 |
> | API `holiday=true` 且 `wage=3` | 2 | 法定节假日 |
> | API `holiday=true` 且 `wage=2` | 3 | 调休假期（周末挪过来的休息日） |
>
> **加班工资倍率判定规则**：
>
> | day_type | wage | 加班工资倍率 |
> |----------|------|-------------|
> | 0 工作日 | 1 | 1.5倍 |
> | 1 休息日 | — | 2倍 |
> | 2 法定假 | 3 | 3倍 |
> | 3 调休假 | 2 | 2倍 |
>
> **使用说明**：
> - 每年初从 timor.tech API 拉取当年数据，自动写入 work_calendar 表（`source='api'`, `confirmed=0`）
> - 系统管理员(role=4)人工确认后 `confirmed=1`，确认后的数据才参与工资计算
> - 未在 work_calendar 表中的日期：周六日按 day_type=1（2倍），周一~周五按 day_type=0（1.5倍）
> - HR/系统管理员可手动增补/修正记录（`source='manual'`）

#### `social_insurance_config`（社保参数配置表）

```sql
CREATE TABLE social_insurance_config (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '配置ID',
  emp_id BIGINT NOT NULL COMMENT '员工ID',
  social_insurance_base DECIMAL(10,2) DEFAULT 0 COMMENT '社保缴费基数（为0则按基本工资）',
  housing_fund_base DECIMAL(10,2) DEFAULT 0 COMMENT '公积金缴费基数（为0则按基本工资）',
  pension_personal DECIMAL(5,2) DEFAULT 8.00 COMMENT '养老保险个人缴费比例(%)',
  pension_company DECIMAL(5,2) DEFAULT 16.00 COMMENT '养老保险公司缴费比例(%)',
  medical_personal DECIMAL(5,2) DEFAULT 2.00 COMMENT '医疗保险个人缴费比例(%)',
  medical_company DECIMAL(5,2) DEFAULT 8.00 COMMENT '医疗保险公司缴费比例(%)',
  unemployment_personal DECIMAL(5,2) DEFAULT 0.50 COMMENT '失业保险个人缴费比例(%)',
  unemployment_company DECIMAL(5,2) DEFAULT 0.50 COMMENT '失业保险公司缴费比例(%)',
  injury_company DECIMAL(5,2) DEFAULT 0.50 COMMENT '工伤保险公司缴费比例(%)',
  maternity_company DECIMAL(5,2) DEFAULT 1.00 COMMENT '生育保险公司缴费比例(%)',
  housing_fund_personal DECIMAL(5,2) DEFAULT 12.00 COMMENT '住房公积金个人缴费比例(%)',
  housing_fund_company DECIMAL(5,2) DEFAULT 12.00 COMMENT '住房公积金公司缴费比例(%)',
  remark VARCHAR(200) COMMENT '备注说明',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='社保参数配置表';
```

> **社保计算说明**：
> - 缴费基数 = base_salary（基本工资），但需在 [min_base, max_base] 范围内封顶保底
> - 个人扣款 = 基数 × (pension_personal + medical_personal + unemployment_personal + housing_fund_personal)%
> - 公司缴纳 = 基数 × (pension_company + medical_company + unemployment_company + injury_company + maternity_company + housing_fund_company)%
> - 默认值按国家常见比例设置，HR 可根据当地政策调整

#### `group_performance`（组绩效表，PM设置，组共享）

> **设计说明**：
> - 绩效由PM设置，同一个PM下属的组员共享同一个绩效金额
> - 每月由PM设置一次，设置后组内所有员工（PG/PL）的performance_bonus统一为此值
> - HR也可以调整（覆盖PM的设置）

```sql
CREATE TABLE group_performance (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '绩效配置ID',
  pm_id BIGINT NOT NULL COMMENT 'PM员工ID（关联employee.id）',
  month VARCHAR(7) NOT NULL COMMENT '绩效月份，格式：2026-06',
  performance_amount DECIMAL(10,2) NOT NULL COMMENT '组绩效金额（组内所有员工共享此金额）',
  set_by TINYINT DEFAULT 0 COMMENT '设置人：0PM 1HR',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  UNIQUE KEY uk_pm_month (pm_id, month),
  INDEX idx_pm_id (pm_id),
  INDEX idx_month (month)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='组绩效表';
```

> **使用说明**：
> - PM通过接口设置组绩效后，系统自动更新 employee 表中该PM下属所有员工的 performance_bonus 字段
> - 每月初PM设置上月绩效（考核周期）
> - HR可覆盖PM的设置（set_by=1）

**`POST /api/holiday/sync?year=2026` 说明**：

调用 timor.tech 节假日 API 获取全年365天日期类型数据，自动解析并写入 `work_calendar` 表。每条记录初始 `confirmed=0`（待确认），需系统管理员人工确认后才能参与工资计算。

**timor.tech API 全年日期查询接口**：

| 项目 | 说明 |
|------|------|
| **接口地址** | `http://timor.tech/api/holiday/year/{year}/?type=Y&week=Y` |
| **示例** | `http://timor.tech/api/holiday/year/2026/?type=Y&week=Y` |
| **协议** | 同时支持 HTTP 和 HTTPS |
| **费用** | 完全免费，不限速，无需注册 |
| **数据范围** | 2013年 ~ 2026年（每年随国务院通知更新） |
| **限额** | 节假日 API 10000次/IP/24小时 |

**关键参数说明**：

| 参数 | 值 | 说明 |
|------|---|------|
| `{year}` | 如 `2026` | 指定年份，**末尾必须加斜杠 `/`**（如 `/year/2026/`） |
| `type` | `Y` | **必须带**，否则不返回日期类型信息（无法区分工作日/周末/节日/调休） |
| `week` | `Y` | **必须带**，否则周末日期不出现在返回数据中（无法标记普通周六日） |

**返回数据结构**：

```json
{
  "code": 0,
  "holiday": {
    "01-01": {
      "holiday": true,     // true=法定假, false=调休补班
      "name": "元旦",       // 节日名称或补班名称（如"元旦后补班"）
      "wage": 3,            // 工资倍数基数：1=正常 2=双倍 3=三倍
      "date": "2026-01-01"  // 日期
    },
    "01-04": {
      "holiday": false,     // false=调休补班（要上班）
      "name": "元旦后补班",
      "wage": 1,            // 补班日正常工资
      "after": true,        // true=节后补班, false=节前补班（仅调休有此字段）
      "target": "元旦"       // 补班关联的节日名称（仅调休有此字段）
    }
  },
  "type": {
    "2026-01-01": {
      "type": 2,            // 0=工作日 1=周末 2=法定假 3=调休补班
      "name": "元旦",       // 类型中文名（周一~周日/假期名/某某补班）
      "week": 4             // 周几：1=周一 ... 7=周日
    },
    "2026-01-04": {
      "type": 3,            // 调休补班日
      "name": "元旦后补班",
      "week": 7
    }
  }
}
```

**type 字段与 work_calendar.day_type 映射**：

| timor.tech `type` | 含义 | → `work_calendar.day_type` | → 加班工资倍率 |
|---|---|---|---|
| 0 | 工作日 | 0（工作日） | 1.5倍 |
| 1 | 周末 | 1（休息日） | 2倍 |
| 2 | 法定假 | 2（法定节假日） | 3倍 |
| 3 | 调休补班 | 0（工作日，补班日要上班） | 1.5倍 |

## 四、核心业务逻辑详解

### 4.1 打卡逻辑（多次打卡取极值 + 跨天05:00截断）

```
POST /api/attendance/clock

输入：无额外参数（从JWT取empId，服务端取服务器时间）
输出：打卡成功 + 今日状态概览

步骤：
  now = LocalDateTime.now()
  today = now.toLocalDate()

  ★ 跨天截断规则：凌晨 00:00~05:00 的打卡归属前一天
  after5 = now.getHour() >= 5
  IF !after5:
    today = today.minusDays(1)    ← 凌晨5点前，record_date 归为昨天

  ① 查询 attendance_record WHERE emp_id=? AND record_date=today
     → 无记录：
       INSERT (emp_id, record_date, clock_in_time=now, status=0, absent_hours=0)
       注：凌晨场景下，如果昨天无记录则属于异常打卡，系统标记缺勤

     → 有记录：
       IF clock_in_time IS NULL OR now < clock_in_time:
         clock_in_time = now          ← 更早的时间覆盖
       clock_out_time = now            ← 更晚的时间始终覆盖

  ② 判定状态 + 计算 absent_hours：
       WORK_START = 09:00, WORK_END = 18:00, FULL_HOURS = 8

       lateHours = MAX(0, (clock_in_time - WORK_START) 的小时数)
       earlyHours = MAX(0, (WORK_END - clock_out_time) 的小时数)
       absent_hours = lateHours + earlyHours

       IF clock_in_time IS NULL AND clock_out_time IS NULL:
         status = 3 (缺勤), absent_hours = FULL_HOURS
       ELIF lateHours > 0 AND earlyHours > 0:
         status = 1 (迟到优先), absent_hours = lateHours + earlyHours
       ELIF lateHours > 0:
         status = 1 (迟到), absent_hours = lateHours
       ELIF earlyHours > 0:
         status = 2 (早退), absent_hours = earlyHours
       ELSE:
         status = 0 (正常), absent_hours = 0

  ③ 保存并返回（含 absent_hours）

--- 定时任务：cutoff 时间判定缺勤 ---

  @Scheduled(cron = "0 0 5 * * ?")    ← 每天05:00执行（跨天加班归属后）
  逻辑：
    遍历昨日所有员工：
      查 attendance_record WHERE emp_id=? AND record_date=昨日
      → 无记录：INSERT (status=3, absent_hours=8, remark="系统判定缺勤")
      → 有记录但 clock_out_time IS NULL: UPDATE status=3, absent_hours=计算剩余小时数

  @Scheduled(cron = "0 59 23 * * ?")  ← 每天23:59执行
  逻辑：
    遍历今日所有 clock_out_time IS NULL 的记录：
      UPDATE status=3, absent_hours=计算值, remark="未打下班卡，系统判定缺勤"
```

**关键点**：
- 不区分上班/下班按钮，前端只有一个"打卡"按钮
- 第一次打卡 = 上班时间，之后每次打卡都更新下班时间为最新
- 缺勤小时数由系统自动计算，前端展示时直接读取 `absent_hours` 字段
- cutoff 时间设为 05:00（次日），避免跨天加班被误判为缺勤
- **凌晨00:00~05:00打卡归属前一天record_date**：保证跨天加班的下班时间写入前一天的记录，工时计算 = clock_out - clock_in 完整连续
- **没有加班审批的人凌晨打卡也归属前一天**：如果昨天无打卡记录则是异常，系统标记缺勤
- **前端展示跨天下班时间**：clock_out_time在00:00~05:00且日期≠record_date时，显示"次日 HH:mm"

**打卡边界处理细节**：

实际业务中需考虑的问题：

| 问题 | 处理方式 |
|---|---|
| 迟到标准时间哪里配置？ | 可做 `sys_config` 配置表或 application.yml 配置，存 `work_start_time`、`work_end_time` |
| 周末需要打卡吗？ | 打卡接口里判断是否工作日，周末返回"今日休息，无需打卡" |
| 同一天重复打卡？ | `uk_emp_date` 唯一索引保证一天一条记录，第二次以后只更新 clock_out_time |
| 下班打卡时没上班记录？ | 先查记录，无则返回"请先上班打卡" |
| 跨天打卡（夜班）？ | **凌晨00:00~05:00打卡归属前一天record_date**，clock_out_time如实记录跨天时间（如02:00），前端显示"次日 02:00" |

**v1 简化版工作日判断**：

```java
// 判断是否工作日（含调休工作日判断）
private boolean isWorkday(LocalDate date) {
    // 先查 work_calendar 表（已确认的记录优先）
    WorkCalendar wc = workCalendarMapper.findByDate(date);
    if (wc != null && wc.getConfirmed() == 1) {
        return wc.getDayType() == 0;  // day_type=0 就是工作日
    }
    // 无记录则按周几判断
    DayOfWeek day = date.getDayOfWeek();
    return day != DayOfWeek.SATURDAY && day != DayOfWeek.SUNDAY;
}

// 判断迟到：以上班时间 09:00 为界
private boolean isLate(LocalDateTime clockInTime) {
    return clockInTime.toLocalTime().isAfter(LocalTime.of(9, 0));
}

// 获取加班工资倍率
private double getOvertimeMultiplier(LocalDate date) {
    WorkCalendar wc = workCalendarMapper.findByDate(date);
    if (wc != null && wc.getConfirmed() == 1) {
        switch (wc.getDayType()) {
            case 0: return 1.5;  // 工作日
            case 1: return 2.0;  // 休息日
            case 2: return 3.0;  // 法定假
            case 3: return 2.0;  // 调休假
        }
    }
    // 无记录：周末2倍，其他1.5倍
    DayOfWeek day = date.getDayOfWeek();
    return (day == DayOfWeek.SATURDAY || day == DayOfWeek.SUNDAY) ? 2.0 : 1.5;
}
```

### 4.2 顺序双人审批通用逻辑

> 加班申请 和 打卡变更 共用同一套审批框架。

```
┌─────────────┐    ┌─────────────┐    ┌──────────────┐
│   step=1    │    │   step=2    │    │   step=3     │
│  等 PL 审批  │──→ │  等 PM 审批  │──→ │  完成        │
│             │    │             │    │              │

└─────────────┘    └─────────────┘    └──────────────┘

updateFinalStatus() 伪代码：
  // PL 驳回 或 PM 驳回 → 直接终态
  IF pl_status == 2 OR pm_status == 2:
    final_status = 2 (已驳回)
    step = 3 (完成)
    RETURN

  // PL 通过 且 PM 还没审 → 等 PM
  IF pl_status == 1 AND pm_status == 0:
    final_status = 0 (待审批中)
    step = 2 (进入PM阶段)
    RETURN

  // 双方都通过
  IF pl_status == 1 AND pm_status == 1:
    final_status = 1 (已通过)
    step = 3 (完成)

    // ★ 变更类型特殊处理：自动执行
    IF 是打卡变更申请:

      executed = 1
      executed_time = now()
```

### 4.3 逐级上报（PL/PM 自身申请的特殊处理）

```
当申请人提交加班或变更申请时：

  applicant = 当前登录员工

  CASE 1: 申请人有 PL 且不是自己
    → step=1, pl_status=0, pm_status=0
    → 正常走 PL→PM 流程

  CASE 2: 申请人就是自己的 PL（pl_id == 自己.id 或 pl_id IS NULL）

    → 跳过 PL 和 PM 审批
    → step=3, pl_status=1, pm_status=1, final_status=1

    → 若是变更申请：立即自动执行
```

**请假天数精确计算（扣除周末）：**

```java
import java.time.DayOfWeek;
import java.time.LocalDate;
import java.time.temporal.ChronoUnit;

public class LeaveDayCalculator {

    /**
     * 计算请假天数，扣除周六周日
     */
    public static double calculateWorkDays(LocalDate start, LocalDate end) {
        if (start.isAfter(end)) return 0;

        // 同一天
        if (start.equals(end)) {
            return isWorkday(start) ? 1.0 : 0.0;
        }

        long totalDays = ChronoUnit.DAYS.between(start, end) + 1;
        long workDays = 0;
        LocalDate temp = start;
        for (long i = 0; i < totalDays; i++) {
            if (isWorkday(temp)) {
                workDays++;
            }
            temp = temp.plusDays(1);
        }
        return workDays;
    }

    private static boolean isWorkday(LocalDate date) {
        DayOfWeek day = date.getDayOfWeek();
        return day != DayOfWeek.SATURDAY && day != DayOfWeek.SUNDAY;
    }
}
```

> **提示**：精确处理法定节假日需结合 `work_calendar` 表判断，已确认(`confirmed=1`)的法定假和调休假计入请假天数，初版可以只扣周末。后续可扩展为结合 `work_calendar` 表判断。

### 4.5 实际加班时长计算（系统自动，基于打卡记录）

> **原则：以实际打卡时间为准，申请时长仅作审批参考，工资按实际时长计算。**

```
计算时机：
  ① 员工打下班卡时（实时计算，写入 overtime_request.actual_hours）
  ② 每天 05:00 定时任务（cutoff 后补算，处理漏算情况）

计算逻辑：
  overtime_date = 加班申请里的 overtime_date
  work_end_time = 18:00（可配置）

  查 attendance_record WHERE emp_id=? AND record_date=overtime_date
  → clock_out_time 为空：actual_hours = 0（未打下班卡，不计算加班）
  → clock_out_time <= work_end_time：actual_hours = 0（未加班）
  → clock_out_time > work_end_time：
      raw_hours = (clock_out_time - work_end_time) 换算为小时，保留1位小数
      actual_hours = MIN(raw_hours, apply_hours)
      → 注：也可改为不封顶（actual_hours = raw_hours），按公司制度二选一

  同时判定工资倍数 salary_multiplier：
    先查 work_calendar WHERE date = overtime_date AND confirmed = 1
    → 有记录（已确认）：
        IF day_type = 0: salary_multiplier = 1.5  ← 工作日加班
        IF day_type = 1: salary_multiplier = 2.0  ← 休息日加班
        IF day_type = 2: salary_multiplier = 3.0  ← 法定假加班
        IF day_type = 3: salary_multiplier = 2.0  ← 调休假加班
    → 无记录或未确认：
        IF 周六或周日：salary_multiplier = 2.0
        ELSE：          salary_multiplier = 1.5

  更新：overtime_request SET actual_hours=?, salary_multiplier=? WHERE id=?
```

**举例**：
```
申请：2026-06-29（周日）加班，apply_hours=4（预期18:00~22:00）
实际：clock_out_time = 20:00（只加到20点）
→ actual_hours = MIN(2.0, 4) = 2.0
→ salary_multiplier = 2.0（周日）
→ 加班费 = 2.0 × 时薪 × 2.0
```

### 4.6 工资计算逻辑

```
缴费基数判定（员工级可单独设置，管理部在新增/调整员工时设置）：
  si_base = employee.social_insurance_base（社保缴费基数）
  IF si_base == 0: si_base = base_salary（为0则按基本工资）
  IF si_base < social_insurance_config.min_base:  si_base = min_base
  IF si_base > social_insurance_config.max_base:  si_base = max_base

  hf_base = employee.housing_fund_base（公积金缴费基数）
  IF hf_base == 0: hf_base = base_salary（为0则按基本工资）
  IF hf_base < social_insurance_config.min_base:  hf_base = min_base
  IF hf_base > social_insurance_config.max_base:  hf_base = max_base

公积金比例判定（员工级可单独设置，为NULL则按全局配置）：
  personal_ratio = employee.housing_fund_ratio_personal
  IF personal_ratio IS NULL: personal_ratio = social_insurance_config.housing_fund_personal
  
  company_ratio = employee.housing_fund_ratio_company
  IF company_ratio IS NULL: company_ratio = social_insurance_config.housing_fund_company

个人社保扣款（按社保基数计算）：
  personal_insurance = si_base × (pension_personal + medical_personal
                             + unemployment_personal) / 100
  personal_housing_fund = hf_base × personal_ratio / 100
  personal_total = personal_insurance + personal_housing_fund

公司社保缴纳（按社保基数计算，仅统计，不从员工工资扣除）：
  company_insurance = si_base × (pension_company + medical_company
                            + unemployment_company + injury_company
                            + maternity_company) / 100
  company_housing_fund = hf_base × company_ratio / 100
  company_total = company_insurance + company_housing_fund

应发工资构成：
  可扣工资（参与缺勤/请假扣款）：
    deductible = base_salary + position_salary
               + housing_subsidy + car_subsidy + meal_subsidy + other_subsidy
    （不含 performance_bonus，绩效不参与任何扣款）

  固定发放（不参与扣款）：
    + performance_bonus（绩效奖金，由PM设置，组共享）

当月实际工作日天数计算：
  work_days = 当月总天数
            - 周六日天数（查 work_calendar 无记录的周六日）
            - 法定节假日天数（work_calendar 中 day_type=2 且 confirmed=1 的天数）
            - 调休假期天数（work_calendar 中 day_type=3 且 confirmed=1 的天数）
            + 调休工作天数（work_calendar 中 day_type=0 且 confirmed=1 且 holiday=0 的补班天数）

请假扣款公式：
  日工资 = deductible / work_days
  事假扣款 = 事假天数 × 日工资
  病假扣款 = 病假天数 × 日工资 × 0.2（按80%发，即扣20%）

缺勤扣款公式：
  时薪 = deductible / work_days / 8
  缺勤扣款 = absent_hours × 时薪

最终应发工资：
  应发工资 = deductible + performance_bonus + 加班费
           - 缺勤扣款 - 请假扣款 - personal_total（个人社保+公积金扣款）

  公司成本 = 应发工资 + company_total（公司社保+公积金缴纳）
```

> **社保/公积金基数说明**：
> - 管理部新增员工时可设置 `social_insurance_base` 和 `housing_fund_base`
> - 若设置为0，则自动按 `base_salary` 计算
> - 基数需在 [min_base, max_base] 范围内封顶保底
>
> **公积金比例说明**：
> - 管理部可设置员工级公积金比例（`housing_fund_ratio_personal` / `housing_fund_ratio_company`）
> - 若员工级比例为NULL，则使用 `social_insurance_config` 表中的全局配置

**月工资计算定时任务**：
```
每月 2 日 03:00 执行：
  @Scheduled(cron = "0 0 3 2 * ?")

  逻辑：
      遍历所有在职员工（status=1, deleted=0）：
        month = 上月（如 2026-06）
        year = month的前4位,  month的后2位

        // 1. 计算当月实际工作日天数
        work_days = 计算当月实际工作日(work_days)

        // 2. 计算可扣工资基数
        deductible = base_salary + position_salary
                   + housing_subsidy + car_subsidy + meal_subsidy + other_subsidy

        // 3. 缺勤扣款
        查上月 attendance_record：total_absent_hours = ∑absent_hours
        hourly_rate = deductible / work_days / 8
        absent_deduction = total_absent_hours × hourly_rate

        // 4. 加班费
        查上月 overtime_request（final_status=1）：
          total_overtime_pay = ∑(actual_hours × hourly_rate × salary_multiplier)

        // 5. 请假扣款
        查上月 leave_request（final_status=1）：
          personal_leave_days = ∑days WHERE leave_type=1
          sick_leave_days = ∑days WHERE leave_type=2
          daily_rate = deductible / work_days
          leave_deduction = personal_leave_days × daily_rate
                          + sick_leave_days × daily_rate × 0.2

        // 6. 个人社保扣款
        查 social_insurance_config WHERE config_year = year
        base = MAX(min_base, MIN(max_base, base_salary))
        personal_insurance = base × (pension_personal + medical_personal
                                   + unemployment_personal + housing_fund_personal) / 100

        // 7. 汇总
        gross_salary = deductible + performance_bonus + total_overtime_pay
        actual_salary = gross_salary - absent_deduction - leave_deduction - personal_insurance

        company_cost = actual_salary
                     + base × (pension_company + medical_company + unemployment_company
                             + injury_company + maternity_company + housing_fund_company) / 100

        写入 salary_record 表（status=0 草稿，HR 确认后改 1）
```

#### JWT 工具类

```java
// common/JwtUtils.java
@Component
public class JwtUtils {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private Long expiration;

    // 生成 token
    public String generateToken(Long empId, String username, Integer role) {
        return Jwts.builder()
                .setSubject(String.valueOf(empId))
                .claim("username", username)
                .claim("role", role)
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(Keys.hmacShaKeyFor(secret.getBytes()), SignatureAlgorithm.HS256)
                .compact();
    }

    // 从 token 解析员工ID
    public Long getEmpIdFromToken(String token) {
        return Long.valueOf(Jwts.parserBuilder()
                .setSigningKey(secret.getBytes())
                .build()
                .parseClaimsJws(token)
                .getBody()
                .getSubject());
    }

    // 验证 token 是否有效
    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder().setSigningKey(secret.getBytes()).build()
                    .parseClaimsJws(token);
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
}
```

#### JWT 拦截器

```java
// config/JwtInterceptor.java
@Component
public class JwtInterceptor implements HandlerInterceptor {

    @Autowired
    private JwtUtils jwtUtils;

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 跳过登录接口
        String uri = request.getRequestURI();
        if (uri.contains("/auth/login")) {
            return true;
        }

        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            response.setStatus(401);
            response.getWriter().write("{\"code\":401,\"message\":\"未登录\"}");
            return false;
        }

        String token = authHeader.substring(7);
        if (!jwtUtils.validateToken(token)) {
            response.setStatus(401);
            response.getWriter().write("{\"code\":401,\"message\":\"token已失效\"}");
            return false;
        }

        // 把员工ID放入请求属性
        Long empId = jwtUtils.getEmpIdFromToken(token);
        request.setAttribute("empId", empId);
        return true;
    }
}
```

```javascript
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: 'http://localhost:8080/api',
  timeout: 10000

// 请求拦截：自动带 token
request.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 响应拦截：统一错误处理
request.interceptors.response.use(
  res => {
    if (res.data.code !== 200) {
      ElMessage.error(res.data.message)
      return Promise.reject(res.data.message)
    }
    return res.data
  },
  err => {
    if (err.response?.status === 401) {
      localStorage.removeItem('token')
      window.location.href = '/login'
    }
    ElMessage.error(err.message)
    return Promise.reject(err)
  }
)
```
